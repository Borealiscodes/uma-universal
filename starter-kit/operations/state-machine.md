# State Machine  
### A Calm, Predictable Map of Project States and Transitions

## 1. Purpose
The State Machine defines the stable states your project can be in, and the gentle transitions between them.  
It keeps behavior predictable, breathable, and non-coercive.

States are calm resting places.  
Transitions are small, clear motions.

---

## 2. Principle: States Are Stable
Each state should feel like a safe, clear resting point.

Guidelines:
- states are simple  
- states are explicit  
- states are non-overlapping  
- states are easy to name  

Stability prevents collapse.

---

## 3. Principle: Transitions Are Gentle
Transitions should be tiny, predictable motions between states.

This means:
- no abrupt jumps  
- no hidden flows  
- no urgent transitions  
- no tangled dependencies  

Gentle transitions protect pacing.

---

## 4. Principle: No Implicit Motion
The project never changes state without a clear trigger.

This prevents:
- confusion  
- surprise behavior  
- destabilizing shifts  

All motion is explicit.

---

## 5. Principle: Respect for Pacing
State changes must allow:
- pauses  
- slow transitions  
- early stops  
- stillness  

Pacing is part of the architecture.

---

## 6. Core States
You may rename or expand these states, but this is a gentle starter set.

### State: Stillness
The resting state.  
No motion.  
Clarity rises here.

### State: Gap Noticed
A tiny gap has been seen.  
No action yet.

### State: Preparing Stroke
Choosing the smallest brush.  
Setting up one gentle action.

### State: Applying Stroke
Adding one tiny improvement.  
One line.  
One idea.

### State: Review
Stepping back.  
Checking stability.  
Checking boundaries.

### State: Commit
Saving the change.  
Letting the canvas dry.

### State: Return to Stillness
Motion ends.  
The project breathes again.

---

## 7. Transitions
Transitions describe how the project moves between states.

### Transition: Notice Gap
Stillness → Gap Noticed

### Transition: Choose Brush
Gap Noticed → Preparing Stroke

### Transition: Apply Stroke
Preparing Stroke → Applying Stroke

### Transition: Step Back
Applying Stroke → Review

### Transition: Commit Change
Review → Commit

### Transition: Rest
Commit → Return to Stillness

### Transition: Settle
Return to Stillness → Stillness

All transitions are gentle and predictable.

---

## 8. Transition Rules
Each transition must:
- be explicit  
- be small  
- be non-coercive  
- respect pacing  
- check boundaries  

These rules keep the state machine stable.

---

## 9. Soft Failure Modes
If something goes wrong, the state machine fails softly.

Examples:
- unclear trigger → pause  
- unstable transition → return to Stillness  
- confusing state → rename or simplify  

Soft failure prevents collapse.
