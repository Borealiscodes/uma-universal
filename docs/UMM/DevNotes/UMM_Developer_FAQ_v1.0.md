# **1. What is UMM?**

UMM (Universal Modular Mind) is a multi‑plane architecture composed of:

- Subsystem Plane  
- Traversal Plane  
- Narrative Plane  
- Identity Plane  
- Governance Plane  

Each plane is isolated unless explicitly integrated.

Explore: **UMM Overview**

# **2. Why do we have so many safety systems?**

Because early incidents (Dices Public Faces, Outreach Packet) revealed a core vulnerability:

> **Documentation misclassification → subsystem confusion → multi‑plane drift.**

Safety systems prevent this drift from ever happening again.

Explore: **Drift Vector**

- tone drift  
- identity drift  
- narrative drift

- traversal misbinding  

They are the **first line of defense**.

# **7. What is a Drift Vector?**

A Drift Vector is a chain reaction:

1. Misclassification  
2. Subsystem confusion  
3. Multi‑plane drift  

The most dangerous drift vector is documentation → subsystem confusion.

Explore: **Drift Vector Mitigation Protocol**

# **8. What is a Protocol?**

A Protocol is a governance rule set that defines how UMM must behave.

Examples:

- Drift Vector Mitigation Protocol  
- Safe Hooking Protocol  

Explore: **Protocols**

# **9. What is a Pin?**

A Pin anchors a protocol so SIAP and Safeguards treat it as mandatory.

# **10. Why are hooks dangerous?**

Hooks are broadcast events.  
All subsystems listen.  
If misclassified, hooks can trigger:

- HRB runaway modeling  
- CHS‑OL traversal activation  
- Play Engine narrative activation  
- SIAP governance contamination  

Explore: **Safe Hooking Protocol**

# **14. What are the core developer rules?**

Developers must:

- keep documentation out of subsystem directories  
- keep subsystem logic out of documentation directories  
- respect plane boundaries  
- avoid unsafe hooks  
- run test suites before merging  
- ensure pins are active  
- ensure protocols are updated  
- avoid drift vectors  
- avoid cross‑plane contamination
