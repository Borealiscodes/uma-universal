# 🧭 **1. Purpose of the Governance Charter**

The UMM Governance Charter exists to:

- protect the **structural integrity** of the UMM  
- define **roles and authorities**  
- prevent **conceptual drift**  
- ensure **ND‑accessibility**  
- maintain **architectural coherence**  
- guide **long‑term evolution** of the system  

This charter is the **constitution** of the UMM project.

---

# 🧱 **2. Governance Principles**

All governance decisions must follow these principles:

### **2.1 Structural Integrity**
The architecture must remain internally consistent across:

- planes  
- seasons  
- geometry layers  
- null‑expansion systems  
- Atropus systems  

### **2.2 Reversibility**
All documents must be:

- reversible  
- non‑coercive  
- low‑ambiguity  
- ND‑accessible  

### **2.3 Non‑Interpretation**
Governance must prevent:

- lore  
- narrative meaning  
- psychological framing  
- metaphorical overreach  

UMM is a **navigation architecture**, not a mythology.

### **2.4 Minimalism**
Governance must avoid unnecessary complexity.

---

# 🧩 **3. Governance Roles**

There are three constitutional roles:

---

## **3.1 Contributors**

Contributors may:

- propose new documents  
- refine existing documents  
- add diagrams, maps, codices, schematics  
- improve clarity and structure  

Contributors may **not**:

- alter invariants  
- redefine planes  
- change seasonal logic  
- modify null‑expansion rules  
- alter Atropus constraints  

---

## **3.2 Maintainers**

Maintainers:

- enforce invariants  
- review PRs  
- ensure correct placement  
- maintain naming conventions  
- prevent conceptual drift  
- manage versioning  
- protect Winter / Null / Atropus strictness  

Maintainers may **reject** contributions that violate the architecture.

---

## **3.3 Governance Stewards**

Stewards are the highest authority.

They may:

- approve MAJOR version changes  
- add new subsystems  
- modify constitutional rules  
- restructure the repo  
- define new invariants  

Stewards may **not**:

- violate ND‑accessibility  
- introduce narrative meaning  
- collapse subsystem boundaries

# 🔒 **5. Constitutional Boundaries**

The following boundaries are **immutable** unless changed by Stewards:

### **5.1 Plane Boundaries**
Map, World, Character, Play Tree, Center must remain distinct.

### **5.2 Seasonal Boundaries**
Spring, Summer, Autumn, Winter each have:

- unique pacing  
- unique collapse/expansion rules  
- unique plane pairings  

### **5.3 Null‑Expansion Boundaries**
Null‑Expansion must remain:

- zero‑growth  
- zero‑texture  
- zero‑drift  
- zero‑accumulation  

### **5.4 Atropus Boundaries**
Atropus must remain:

- anchor‑only  
- tone‑minimal  
- branchless  
- narrative‑free  

These boundaries define the **constitutional shape** of UMM.

---

# 🧪 **6. Governance Review Process**

Every proposed change must undergo:

### **6.1 Structural Review**
Does it violate:

- invariants  
- plane logic  
- seasonal logic  
- null‑expansion rules  
- Atropus constraints  

### **6.2 Placement Review**
Is the file in the correct folder?

### **6.3 Naming Review**
Does the filename follow:

```
UMM-[SCOPE]-[SUBJECT].md
```

### **6.4 Link Review**
Are Guided Links present and correct?

### **6.5 Diagram Review**
Are ASCII diagrams structurally accurate?

If any fail → revision required.

---

# 🧭 **7. Version Governance**

UMM uses **semantic conceptual versioning**:

```
MAJOR.MINOR.PATCH
```

### **MAJOR** — structural changes  
Requires Steward approval.

### **MINOR** — new documents  
Requires Maintainer approval.

### **PATCH** — corrections  
Can be approved by any Maintainer.
