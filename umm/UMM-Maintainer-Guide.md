# 🧭 **1. Maintainer Role Definition**

Maintainers are responsible for:

- **architectural coherence**  
- **structural integrity**  
- **version governance**  
- **document lineage**  
- **boundary enforcement**  
- **reviewing contributor changes**  
- **ensuring ND‑accessibility**  
- **preventing conceptual drift**  

Maintainers are *not* responsible for:

- interpreting UMM  
- expanding UMM lore  
- adding narrative meaning  
- psychological framing  

UMM is a **navigation system**, not a mythology.

---

# 🧱 **2. Architectural Integrity Rules**

Maintainers must enforce:

### **2.1 No Cross‑Plane Drift**
A file must not mix:

- Map logic with World logic  
- Character logic with Play Tree logic  
- Center logic with narrative logic  

If a file crosses planes, it must be split.

### **2.2 No Seasonal Leakage**
Spring, Summer, Autumn, Winter each have:

- their own pacing  
- their own boundaries  
- their own collapse/expansion rules  

A Winter file cannot contain Summer logic.

### **2.3 No Null‑Expansion Violations**
Null‑Expansion files must obey:

- zero‑growth  
- zero‑texture  
- zero‑drift  
- zero‑accumulation  

If a contributor adds expansion to a null file, reject it.

### **2.4 Atropus Strictness**
Atropus files must remain:

- anchor‑only  
- tone‑minimal  
- branchless  
- narrative‑free  

Atropus is the strictest subsystem.

---

# 🧩 **3. Versioning Rules**

UMM uses **semantic conceptual versioning**:

```
MAJOR.MINOR.PATCH
```

### **MAJOR** — structural change  
Examples:

- new plane  
- new seasonal invariant  
- new geometry layer  

### **MINOR** — new documents or expansions  
Examples:

- new map  
- new codex  
- new atlas  

### **PATCH** — corrections  
Examples:

- typo fixes  
- formatting  
- link corrections  

Maintainers decide when a change is MAJOR.

---

# 🗂️ **4. Folder Governance**

Maintainers must ensure:

### **4.1 Correct Placement**
Use the rule:

> Place the file in the *lowest folder* that fully describes its function.

### **4.2 No Duplicate Concepts**
If a contributor proposes a file that duplicates an existing concept:

- merge  
- consolidate  
- or reject  

### **4.3 No Orphan Files**
Every file must belong to:

- a subsystem  
- a season  
- a geometry layer  
- or a null‑plane cluster  

If it doesn’t, it must be moved or removed.

---

# 🧪 **5. Review Checklist**

Before approving a PR, check:

- **UMM Invariants** respected  
- **UMM Planes** not violated  
- **Yearly Structure** consistent  
- **Seasonal System** aligned  
- **Null‑Expansion Architecture** obeyed  
- **Atropus Null‑Plane** respected  
- **Guided Links** present  
- **ASCII diagrams** correct  
- **Naming conventions** followed  
- **Folder placement** correct  

If any fail → request changes.

---

# 🧊 **6. Winter / Null / Atropus Enforcement**

These subsystems require special enforcement.

### **Winter**
- stillness pacing  
- collapse vectors  
- no expansion  

### **Null‑Expansion**
- zero‑growth  
- zero‑texture  
- zero‑drift  

### **Atropus**
- anchor‑only  
- tone‑minimal  
- no branching  
- no narrative buildup  

Maintainers must reject any PR that violates these.

---

# 🧼 **7. Style & Tone Enforcement**

Maintainers ensure:

- ND‑friendly pacing  
- low ambiguity  
- low branching  
- reversible structure  
- symbolic, not interpretive language  

Reject if:

- metaphorical overreach  
- psychological claims  
- narrative framing  
- lore‑building  
- coercive tone  

UMM is **not** a story universe.

---

# 🔗 **8. Guided Link Requirements**

Every document must include:

- links to parent systems  
- links to sibling systems  
- links to deeper layers  
- links to operational tools  

Maintainers must check for:

- broken links  
- missing links  
- incorrect link targets

# 🧭 **9. Maintainer Workflow**

1. Read the PR  
2. Identify subsystem  
3. Validate invariants  
4. Validate placement  
5. Validate naming  
6. Validate structure  
7. Validate links  
8. Validate diagrams  
9. Approve or request changes  

This workflow protects the architecture.
