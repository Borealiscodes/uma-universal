## ⭐ 2. The Core Phenomenon  
Two registries exist:

- **Roots Registry** — the constitutional ledger  
- **Pin Registry** — the operational index  

When **PIN‑084** was introduced as a **Protocol‑Embedded Pin**, Roots Registry needed a **structural version increment** (3.6 → 3.7).  
But the Pin Registry did **not** need a structural increment — only a **content reflection**.

Thus:

- Roots → **v3.7**  
- Pin Registry → **still v2.4**, but needing regeneration to *mirror* the new PEP

The system offered **v2.4 regeneration** because:

- Pin Registry versioning is **independent**  
- Pin Registry increments only when its **schema** changes  
- Adding a PEP does **not** change its schema  
- Therefore, the correct next action is **regenerate v2.4**, not increment it

This is expected behavior — but it *feels* anomalous because PEPs create cross‑registry ripples.

---

## ⭐ 3. Why Roots Incremented but Pin Registry Didn’t  
Roots Registry increments when:

- a new pin class is introduced  
- adjacency mapping changes  
- continuity guarantees expand  
- governance lineage shifts  
- a new POSITION taxonomy is added  

PEP‑084 triggered **all five**.

Pin Registry increments only when:

- the registry format changes  
- the registry schema changes  
- the registry tiering changes  

PEP‑084 triggered **none**.

Thus:

- Roots → **structural change → v3.7**  
- Pin Registry → **content update → regenerate v2.4**

This is the architectural split.

---

## ⭐ 4. What the PEP Actually Did  
The introduction of **Protocol‑Embedded Pins** created a new governance behavior:

### **PEPs unify protocol identity and pin identity.**  
This means:

- the protocol defines the pin  
- the registry mirrors the protocol  
- the pin cannot be separated from the protocol  
- the protocol becomes a governance object  
- the registry becomes a continuity object  

This is a **major architectural shift**.

### **PEPs create dual‑registry obligations.**  
Every PEP must:

- live inside the protocol  
- be mirrored in Roots  
- be reflected in the Pin Registry  
- be tracked by the Protocol Lifecycle Engine  
- be anchored by Boundary Layer v2.0  

This creates **registry divergence** — not conflict, but specialization.

### **PEPs introduce a new POSITION lineage.**  
PIN‑084’s POSITION:

```
5D-EXD-SK-PRT-01
```

This is a new branch in the Roots taxonomy, requiring a version increment.

Pin Registry does not track POSITION taxonomy — so no increment.

---

## ⭐ 5. Why the System Offered “Regenerate v2.4”  
Because the Pin Registry:

- did not need a version bump  
- did need to reflect the new PEP  
- did need to update its index  
- did need to maintain continuity with Roots  
- did need to mirror the canonical path  

Thus the correct action is:

**Regenerate v2.4, not increment it.**

This is the same behavior you see in:

- CHS Core updates  
- SIAP Spine updates  
- Boundary Layer updates  

Where Roots increments, but Pin Registry simply regenerates.

## ⭐ 6. The Emergent Behavior: Registry Divergence  
Introducing PEPs created a new emergent pattern:

### **Roots Registry evolves structurally.**  
It absorbs:

- new pin classes  
- new governance behaviors  
- new adjacency rules  
- new continuity guarantees  

### **Pin Registry evolves reflectively.**  
It mirrors:

- pin existence  
- pin paths  
- pin status  
- pin tier  

This divergence is **healthy** — it prevents governance bleed.
