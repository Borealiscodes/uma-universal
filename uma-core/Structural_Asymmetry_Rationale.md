**Version:** 1.0

## Purpose
This document explains why the UMA Base Architecture is intentionally asymmetrical. The asymmetry is a functional requirement derived from constitutional principles, engineering constraints, runtime behavior, and cognitive safety. It must not be altered for aesthetic symmetry.

## Overview
UMA is a load‑bearing cognitive architecture. Its organs, subsystems, and runtime flows do not carry equal weight, operate at equal frequency, or share identical roles. As a result, the architecture cannot be symmetrical without compromising stability, flexibility, and truth preservation.

Asymmetry is a structural feature, not an artifact.

## Constitutional Basis
The Five‑Point Core (Warmth, Conscience, Awareness, Truth, Humility) produces directional constraints. These constraints generate non‑uniform governance behavior across the constitutional organs:

- Roots require central proximity for timeline anchoring.
- Pins require offset positioning for milestone differentiation.
- Influence Map requires lateral placement for cause‑effect mapping.
- Timestamps require lower‑ring placement for receipt accumulation.
- Audit Trails require bottom‑center placement for truth verification.
- RFCs require offset placement for governance flexibility.

These roles cannot be evenly distributed without violating constitutional logic.

## Engineering Basis
The Engineering Exoskeleton is designed for functional load distribution:

- Substrate Stack anchors storage above the Core.
- Orbital Markers provide lateral reference points.
- Engines operate below the Core for downward runtime flow.
- Runtime Clock sits left‑of‑center for pacing control.
- Safety Boundaries sit right‑of‑center for protective gating.
- Interface Layer sits left‑upper for communication ingress.

These placements reflect runtime frequency, load paths, and safety boundaries. Symmetry would misrepresent engineering reality.

## Runtime Basis
The Imagination Runtime is intentionally unbalanced:

- Generative function sits above the Core.
- Simulation function sits right‑of‑center.
- Integration function sits below the Core.

This reflects directional flow: generation → simulation → integration. Symmetry would imply equal runtime distribution, which is false.

## Information Flow Basis
The constitutional information flow is directional:

Input → Roots → Timestamps → Pins → Influence Map → Audit Trails → RFCs → Roots (updated)

Directional flow cannot be expressed symmetrically without losing clarity or accuracy.

## Humility Principle
Humility (the fifth Core principle) requires:

- non‑rigidity  
- non‑totalizing geometry  
- non‑ornamental structure  
- provisional topology  
- adaptive expansion  

Perfect symmetry would violate Humility by implying completeness, rigidity, and finality.

## Non‑Duality Principle
Non‑duality in UMA is expressed through integration, not mirroring. Integrated systems produce functional asymmetry. Symmetry would imply dualistic separation or ornamental design.

## Cognitive Safety Basis
Symmetry can imply:

- equal weight  
- equal authority  
- equal runtime frequency  
- equal governance priority  

These implications are false and can create cognitive distortion. Asymmetry preserves accurate cognitive mapping.

## Architectural Conclusion
Asymmetry is:

- constitutional  
- structural  
- functional  
- runtime‑accurate  
- cognitively safe  
- humility‑aligned  
- non‑dual  
- load‑bearing  

It must remain part of the UMA architecture.

## Modification Policy
- Asymmetry must not be “corrected” into symmetry.
- Adjustments must preserve functional topology.
- Any proposed changes require an RFC and stress testing.
- Symmetry‑driven edits will be rejected.

## Canonical Statement
**Asymmetry in UMA is intentional, functional, and required. It is a structural expression of constitutional principles, engineering constraints, runtime behavior, and cognitive safety.**
