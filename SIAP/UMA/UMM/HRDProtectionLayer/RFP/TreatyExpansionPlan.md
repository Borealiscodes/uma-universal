## **1. Purpose**
Define the **governance, structure, and lifecycle** for expanding the Rights‑Flagging Protocol (RFP) to support **all major international treaty systems**, including:

- **UN**  
- **ECHR**  
- **OAS**  
- **CCJ / CARICOM**  
- (future) **ACHPR**, **ECOWAS**, **ASEAN**, **Commonwealth**, etc.

This plan ensures the HRD Protection Layer can:

- detect violations  
- classify harms  
- map jurisdiction  
- recommend forums  
- escalate remedies  
- support litigation strategy  
- pass SIAP audits  
- remain extensible for decades  

It is the **root artifact** for all future treaty repositories.

---

## **2. Expansion Philosophy**
The RFP must remain:

- **Modular** — each treaty family is isolated  
- **Composable** — repositories can be combined  
- **Auditable** — SIAP can validate each module  
- **Extensible** — new treaties can be added without redesign  
- **Non‑derogable** — core protections cannot be overwritten  
- **Traversal‑safe** — CHS‑OL nodes bind cleanly  

This ensures long‑term stability.

---

## **3. Repository Architecture**
All treaty families follow the same structure:

```
/RFP/TreatyRepository/
    UN/
    ECHR/
    OAS/
    CCJ/
    (future)
    ACHPR/
    ECOWAS/
    ASEAN/
    Commonwealth/
```

Each repository contains:

```
metadata/        # treaty text, articles, definitions
violations/      # triggers, thresholds, patterns
jurisdiction/    # who can hear what, when
remedies/        # available outcomes
escalation/      # precautionary measures, interim relief
mapping/         # links to FRE, CAC-Map, Matrix
tests/           # SIAP audit hooks
```

This structure is **immutable** and applies to every treaty family.

## **4. Expansion Order (Structurally Optimal)**

### **Phase 2 — ECHR Repository**

### **Phase 4 — CCJ Repository**
Caribbean constitutional obligations (CARICOM, CCJ).

This order ensures:

- universal → regional → constitutional  
- no jurisdiction collisions

- stable SIAP audits
